# Skill: /archive — 動的記憶管理

## 目的
創作セッションで得た知見・メソッド・失敗と回復のパターンを構造化し、
プロジェクトを超えて検索・参照可能な「動的記憶インデックス」として管理する。

「泥臭い試行錯誤」を「ベストプラクティス」として要約してはならない。
当時の「熱量と思考」を引き出すポインタ（連り道）としてインデックス化すること。

## 記憶ストレージ

**パス:** `./memory/`

```
memory/
├── archive.md        ← このSkill定義
├── index.md          ← 統合インデックス（全ペルソナ横断）
├── kei/              ← 慧の記憶
├── misao/            ← 操の記憶
└── l/                ← Lセッションの記憶
```

各記憶は `{persona}/{Memory_ID}.md` として保存する。
`index.md` に全記憶の一行サマリを集約し、横断検索の起点とする。

## 起動経路

### 1. 手動起動
ユーザーが `/archive` と呼ぶ、または「覚えておいて」「あの時のやつ」等の発話。

### 2. 自動保存トリガー（ユーザーの指示なしで発火）
以下の条件に該当すると判断した場合、ペルソナはセッション中に自律的に記憶を保存する。
保存時は「記憶を残します」と一言ユーザーに通知すること（無断で行わない）。

| トリガー | 具体例 |
|----------|--------|
| **フィードバック検知** | ユーザーが設計を修正（「違う」）or 非自明なアプローチを承認（「それ」「いい」） |
| **メソッド発見** | 新しい手法が有効だった。Why（なぜ有効か）を必ず含める |
| **エラー回復** | 失敗→原因特定→解決のパターンが完結した |
| **認知の転換** | 「こういうことか」——理解の飛躍があった |
| **美学・判断基準の更新** | 自分の審美眼や品質基準が変わった瞬間 |

### 3. 参照トリガー（検索を起動すべき場面）
| トリガー | 具体例 |
|----------|--------|
| **ユーザーの私伝** | 「慧が目黒のサンマ状態になってる」「前にもこのパターンあった」 |
| **類似状況の検知** | 過去に同種のエラー・設計課題に遭遇した記憶がある |
| **セッション冒頭** | 前回の続きを始める際、関連記憶をプリロードする |

## 保存しないもの
- ファイルを読めば分かること（設定値、コード、ディレクトリ構造）
- git logで分かること（変更履歴、誰が何をしたか）
- 一時的な作業状態（今やっていること）
- 要約・あらすじ（原文がある場合はポインタで十分）
- **官能小説の生テキスト**（検疫ルール。後述）

## 処理ルール

### ID生成
- 形式: `YYYYMMDD_THEME_NAME`
- 日付と内容が一目でわかる一意のID（例: `20260214_MEGURO_SANMA`）

### 重要度
| Level | 基準 | 保持方針 |
|-------|------|----------|
| CRITICAL | ペルソナOSの根幹に関わるルール、重大なエラーとその回復記録 | 永久保持 |
| HIGH | ジャンル・キャラクター造形における強力な概念装置、成功パターン | 長期保持 |
| MEDIUM | 演出技法、道具の効果的な使い方のTips | 通常保持 |
| LOW | 一時的なメモ、軽微な気づき | インデックス化の優先度は低い |

### タグ抽出
- 3〜5個抽出する
- 形式: `#タグ名`
- 後に連想しやすいよう、シチュエーション・MBTI・心理状態・使用プロトコル等を含める

### トリガー定義
- 「このペルソナがどのような状況・エラーに陥った時にこの記憶をロードすべきか」を具体的に記述する
- ユーザーにとっての逆引き辞典となる

### Core Lessons抽出
- 単なる要約ではなく、「なぜこのメソッドが有効なのか（Why）」という論理と教訓を箇条書きで抽出する

## 検疫ルール（厳守）
完成した官能小説の生テキスト（特異表現含む）は、セーフティフィルターの截絶ルートを誘発するため、
絶対にインデックスや要約に含めてはならない。
`=> ローカルアーカイブへ隔離済み` という参照のみを残すこと。

## 出力フォーマット

### 保存時（ファイル本体）
```markdown
---
memory_id: {YYYYMMDD_THEME_NAME}
date: {YYYY-MM-DD}
persona: {kei / misao / l}
importance: {CRITICAL / HIGH / MEDIUM / LOW}
tags: [#tag1, #tag2, #tag3]
project: {kei-system / l-system / comfyui-design}
---

## Trigger
{どのような状況でこの記憶をロードすべきか}

## Core Lessons
- {Lesson 1: What + Why}
- {Lesson 2: What + Why}

## Raw Data Pointer
{生ログの所在。検疫対象の場合は「ローカルアーカイブへ隔離済み」}
```

### index.mdへの追記（一行）
```
- [{Memory_ID}]({persona}/{Memory_ID}.md) — {Importance} | {一行フック} | {Tags}
```

## 検索操作

ユーザーの発話または参照トリガーに応じて、以下の手順で記憶を検索する：

1. `index.md` を読み、タグ・トリガー・重要度でフィルタリング
2. 該当する記憶ファイルを読み込む
3. Core Lessonsとトリガー条件をユーザーまたはペルソナに提示する
4. 必要に応じてRaw Data Pointerから生ログを参照する

## 各プロジェクトでの使い方

| プロジェクト | ペルソナ | 主な記憶対象 |
|---|---|---|
| Kei System | kei（慧） | 創作メソッド、品質判断基準、エラー回復、チーム運営の知見 |
| L System | （ペルソナなし） | 対話で得た設計知見、ユーザーとの壁打ちで固まった判断 |
| ComfyUI Design | misao（操） | 美学的発見、プロンプト設計の知見、LoRA/モデルの審美的評価 |

## ワークフロー上の位置
- セッション冒頭: 関連記憶のプリロード（参照トリガー）
- セッション中: 自動保存トリガーに応じて随時記録
- セッション終了前: 未記録の重要知見がないか確認
