# Affairs — 官能小説執筆事務局

BL・ゲイ官能小説の創作プロセスを、
LLMを活用した事務的手順に分解するフレームワークです。

読みたい小説がないなら、自分で作りましょう。
作るのに才能は要らない。必要なのは手順だけ。怪力乱神を語らずです。

## これは何か

Affairsは、BL・ゲイ官能小説の「自炊」環境です。

自分の好きなニッチジャンルで読みたい小説が供給されない——
よくある話です。
このシステムは、その不足を自分で埋めるための道具立てを提供します。
プロットの発想から詳細設計、執筆まで、創作の各段階をLLMとの協働作業として整理してあります。

### 射程

**対象：**
BL・ゲイ官能小説。
攻め手と受け手の心理設計、身体描写、
尊厳と官能の使い分けといった、
このジャンル固有の問題を扱います。

**対象外：**
官能を含まない一般文芸、商業出版向けの原稿制作も可能ではあります。
感情設計や心理トレースの手法は他ジャンルでも使えますが、そこは各自で工夫したらよろしいでしょう。

## 三つのシステム

Affairsには三つの作業環境があります。
用途が違うだけで、知識基盤（knowledge）は共有しています。

**L** — 対話型の壁打ち環境。
あなたとLLMが一対一で、プロットを練り、設計を固め、執筆に進む。
一番基本的な環境で、迷ったらまずここから始めてください。

**Kei（慧）** — パイプライン型の生産ライン。
プロット・構成・心理設計・身体設計・執筆を、
それぞれ専門のエージェントが分業で処理します。
半自動で作品が完成するので、設計が明確で、量産したいときに向いています。

**Stage** — 演劇型の即興リレー環境。
キャラクターごとにペルソナを持った「役者」がシーンを演じ、
「監督」が複数テイクから素材を選んで編集する。
キャラクターの意外性を引き出したいときに使います。

どれか一つだけ使っても構いません。
LとStageを連携させることもできます。

## 使い方

創作のプロセスを四つのフェーズに分けています。
どのシステムを使うかに関わらず、流れは同じです。

### 各フェーズであなたがやること

| フェーズ | あなたが出すもの | LLMが返すもの |
|---------|----------------|-------------|
| 1. 発想 | 「年上攻め×意地っ張り、再会もの」程度の着想 | プロットの種を複数案 |
| 2. 構成 | どの案で行くかの判断 | 章構成と温度曲線 |
| 3. 設計 | ここで泣かせる、ここで抱かせる | 心理トレースと身体設計 |
| 4. 執筆 | 確定した設計書 | 本文 |

「面白いBL小説を書いてください」では動きません。
あなたが各段階で判断を下し、LLMはその判断を形にします。

### フェーズ1：発想・壁打ち
「こういう話が読みたい」から始めます。
LLMと対話しながら、核になるアイデアを見つける段階です。
ジャンル、関係性の力学、どこに興奮するのか——
漠然としていて構いません。

### フェーズ2：物語構成設計
プロットを組み、章構成とペースを決めます。
温度曲線（どこで上げてどこで落とすか）、
物語の収束方向を設計します。

### フェーズ3：詳細設計
攻め手の支配ロジック、受け手の心理トレース、
体位・施術の身体設計を行います。
ここが一番「事務的」なフェーズです。

### フェーズ4：執筆
確定した設計書をもとに、本文を生成します。
設計がしっかりしていれば、
執筆は最も機械的な工程になります。

## 必要なもの

### LLM環境
以下のいずれかが必要です。全て使う必要はありません。
むしろ、一つで全てを賄おうとしないでください。
理由はすぐにわかります。

- **Claude Opus4.6**（Anthropic） — 構造分析・設計・批評に優れます。
  ただし肝心の濡れ場に急に解像度が下がります。抽象的表現、婉曲表現で押し切る場合のみ使用可能。
  濡れ場は別の子に。無理に書かせると学級委員長が警告メッセージで説教を始めます。
- **Gemini 3.1Pro**（Google） — 本文生成の主力。比較的おおらか。本文も濡れ場もこなせます。
- **Grok 4.20エキスパートモード**（xAI） — さらにおおらか。ご家庭が放任系なので屈託がなく、ねちっこい描写は苦手。
- **ローカルLLM**（LM Studio等） — 誰にも怒られません。ただし環境整備に貢ぐ必要あり。

### ツール
- **Claude Code** — Kei・Stageの運用に使用。ターミナル操作が必要です。
  分からなければClaude Codeに聞けば教えてくれます。

私の環境ではGoogle Drive上にknowledgeを置き、
LM Studioでローカルモデルを動かしていますが、
必須ではありません。当然ながらファイルはローカルに全て置いても動きます。

### 技術的な前提
プログラミングの知識は不要です。
ただしClaude Codeの基本操作（インストール、起動、コマンド入力）は必要です。
分からなければClaude.aiやClaude Codeに聞けば教えてくれます。

```
│   ├── キャラクター設計フレームワーク.md
│   ├── 表現技法体系.md
│   ├── Ero-Plot-Architecture.md
│   ├── voice_taxonomy_v2.md

│   └── Knowledge_Sex_Techniques.md
├── kei/                    ← パイプライン型生産ライン
│   ├── CLAUDE.md
│   ├── kei.md, ota.md, masamichi.md, ...  ← エージェント定義
│   └── .mcp.json
├── L/                      ← 対話型壁打ち環境
│   ├── CLAUDE.md
│   └── skills/
├── stage/                  ← 演劇型即興リレー環境
│   ├── CLAUDE.md
│   ├── skills/
│   ├── personas/
│   └── scripts/
├── memory/                 ← 動的記憶管理（Archivist）
│   ├── archive.md
│   └── index.md
└── examples/               ← サンプル記憶ファイル
    ├── Kei_System_Memory.md
    └── kei_memory_log_01.md
```

knowledgeディレクトリは三つのシステムから共有参照されます。
各システムのCLAUDE.mdが、そのシステムの動作を定義する設計書です。
